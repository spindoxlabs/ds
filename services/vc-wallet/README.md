# ds-vc-wallet

A minimal DCP Credential Service (VC wallet). Each participant runs their own instance. Holds pre-issued Verifiable Credentials and returns them as a Verifiable Presentation when queried by EDC during contract negotiation.

Provider port: `38082`
Consumer port: `38083`
Caddy URLs: `https://vc-wallet-provider.dataspaces.localhost`, `https://vc-wallet-consumer.dataspaces.localhost`

---

## Purpose

During DCP negotiation, the requesting EDC connector must prove membership in the dataspace by presenting a `MembershipCredential` issued by the trust anchor. EDC calls the configured Credential Service URL to retrieve applicable VCs and assembles them into a Verifiable Presentation.

This service is that Credential Service. It serves the participant's pre-issued VCs as a JSON-LD VP in response to DCP presentation queries.

The Credential Service URL is registered in the participant's DID document:

```json
{
  "id": "did:web:provider.dataspaces.localhost#credential-service",
  "type": "CredentialService",
  "serviceEndpoint": "https://vc-wallet-provider.dataspaces.localhost/api/v1"
}
```

---

## Endpoints

### `POST /api/v1/presentations/query`

Returns a `VerifiablePresentation` containing VCs that match the query. The request body is a DCP presentation query with a `presentationDefinition`.

For dev, all held VCs are returned regardless of input descriptor specifics.

Response is `Content-Type: application/ld+json`.

### `GET /api/v1/credentials`

List all Verifiable Credentials held by this participant.

### `GET /api/v1/credentials/{id}`

Retrieve a specific VC by its `id` field.

### `GET /health`

Liveness check.

## Credential storage

VCs are stored as JSON files in the directory configured by `VC_WALLET_CREDENTIALS_PATH`. Each `.json` file is a single Verifiable Credential.

For dev, the directory is `src/credentials/{provider,consumer}/` and contains pre-issued VCs generated by `scripts/issue-vcs.py`.

### Membership credential structure

```json
{
  "@context": [...],
  "id": "urn:uuid:...",
  "type": ["VerifiableCredential", "MembershipCredential"],
  "issuer": "did:web:trust-anchor.dataspaces.localhost",
  "issuanceDate": "...",
  "credentialSubject": {
    "id": "did:web:provider.dataspaces.localhost",
    "memberOf": "https://dataspaces.localhost/dataspace",
    "role": "Provider",
    "allowedScopes": ["dataspaces.query"]
  },
  "proof": {
    "type": "JsonWebSignature2020",
    "verificationMethod": "did:web:trust-anchor.dataspaces.localhost#key-1",
    "jws": "<es256-jwt>"
  }
}
```

The `proof.jws` is an ES256 JWT signed with the trust anchor's private key. Counterparty EDC verifies it by resolving `did:web:trust-anchor.dataspaces.localhost` and checking the `publicKeyJwk`.

## Issuing VCs

Run `scripts/issue-vcs.py` to issue membership VCs for both participants. The script reads the trust anchor's private key from `src/ds/connector/config/trust-anchor-key.json` and writes signed VCs to `src/credentials/{provider,consumer}/membership-vc.json`.

Re-run after any key rotation (`scripts/gen-keys.sh`).

---

## Configuration

All settings use the `VC_WALLET_` prefix:

- `VC_WALLET_PARTICIPANT_DID` — participant DID URI
- `VC_WALLET_CREDENTIALS_PATH` — path to the directory containing VC JSON files

The matching EDC property:
```properties
edc.credential.service.url=http://vc-wallet-provider:8081/api/v1
```

## Known limitations

- `presentations/query` returns all held VCs regardless of the `presentationDefinition` input descriptors. A production wallet should implement full DIF Presentation Exchange matching.
- Credential status list (`StatusList2021`) is referenced in issued VCs but not enforced — revocation is not checked at query time in dev.
